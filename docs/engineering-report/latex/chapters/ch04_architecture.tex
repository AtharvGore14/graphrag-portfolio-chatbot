\paragraph{Contested decisions.}
FinBERT scores financial sentiment of market text; the product requires classifying \emph{user emotion in chat}, so DistilRoBERTa supplies a discrete label and confidence that can be gated before a MoodEvent is persisted---the LLM must not invent mood~\cite{distilroberta,trd}. MVP evidence is a typed holdings graph under \texttt{user\_id}; Cypher returns attributable properties for citations and guardrails, whereas a vector index as the \emph{primary} path would weaken the user-scope security anchor without solving multi-hop portfolio questions (hybrid/embedding retrieval remains Planned/Future, not Implemented)~\cite{prd,trd}. Agent loops add latency and obscure audit trails; the repository implements one retrieval and one generation per turn in application services rather than LangChain/LlamaIndex~\cite{trd}.

\section{Logical and physical architecture}

At runtime a single FastAPI process orchestrates: a \textbf{Client} (static test page or host UI calling \texttt{POST /chat} with \texttt{user\_id}, message, optional \texttt{session\_id}); the \textbf{API process} (routes, lifespan, \texttt{run\_chat\_turn} in \texttt{services/chat.py}); \textbf{Neo4j} (portfolio graph and MoodEvents; all reads start at \texttt{(:User \{id: \$user\_id\})}); \textbf{Redis} (short-term turn window); \textbf{Mood classifier} (DistilRoBERTa in-process at startup); and an \textbf{LLM gateway} (one OpenAI-compatible client when a templated answer is not enough).

\reportfigure{fig_09_logical_architecture.png}%
{Logical component architecture of the Portfolio GraphRAG Chatbot.}%
{fig:logical-arch}%
{Boxes: Web/Client, FastAPI, chat orchestrator, Neo4j, Redis, Mood, LLM.}

On lifespan start the API opens a pooled Neo4j driver and Redis client, runs \texttt{ensure\_schema}, warms the mood classifier (\texttt{mood\_ready}), and closes connections on shutdown~\cite{fastapi}. Chat requests never open per-request drivers. The static test UI mounts after API routes so \texttt{/health} and \texttt{/chat} take precedence~\cite{readme}. Physically, Compose services \texttt{api}, \texttt{neo4j}, and \texttt{redis} match Table~\ref{tab:compose-topo}; the API reaches Neo4j/Redis by Compose DNS; the LLM is egress HTTP; mood weights live in the HF cache volume. There is no separate mood microservice or vector store in MVP. Production may relocate Neo4j/Redis to managed hosts while keeping the same API image (Future Scope).

\section{Chat orchestration}
\label{sec:chat-orch}

Every accepted chat request is handled by \texttt{run\_chat\_turn} in \texttt{backend/app/services/chat.py}:

\begin{enumerate}[leftmargin=*]
  \item \textbf{Route gate.} \texttt{POST /chat} verifies Neo4j connectivity and mood readiness; otherwise HTTP 503.
  \item \textbf{Intent.} \texttt{detect\_intent} classifies the message (deterministic taxonomy).
  \item \textbf{Parallel fan-out.} Mood classification, Redis history load, and portfolio-universe probe (\texttt{user\_has\_portfolio} + ticker list) via \texttt{asyncio.gather}.
  \item \textbf{Ticker resolve / clarify.} Fuzzy or exact resolution may produce a clarification (refused) without the LLM.
  \item \textbf{Retrieve.} \texttt{retrieve\_subgraph} loads a minimal user-scoped context and citation payloads.
  \item \textbf{Answer.} Prefer a deterministic template for common intents (\texttt{count\_holdings}, \texttt{top\_performers}, single \texttt{holding\_lookup}); otherwise \texttt{generate\_grounded\_answer} with guardrails. Templates skip the LLM for those high-frequency facts.
  \item \textbf{Persist.} Append turns to Redis; optionally write a MoodEvent when confidence gates pass.
\end{enumerate}

This realises ``one retrieval, one generation'' while allowing a \emph{zero-LLM} path when templates already ground the answer.

\section{Ingest, health, and source map}

Ingest is offline relative to chat. Operators load portfolio JSON into Neo4j under a chosen \texttt{user\_id}; chat never trusts client-supplied holdings. CLI entry \texttt{scripts/ingest\_portfolio.py} calls \texttt{services/ingest.py}, which validates required top-level keys (\texttt{metrics}, \texttt{sectors}, \texttt{holdings}), ensures schema, and upserts the user-scoped graph idempotently.

\reportfigure{fig_09_ingest_flow.png}%
{Offline ingest flow from portfolio JSON into Neo4j.}%
{fig:ingest-flow}%
{JSON $\rightarrow$ ingest script $\rightarrow$ Neo4j.}

\texttt{GET /health} reports connectivity without a full chat turn: \texttt{neo4j} (Bolt \texttt{verify\_connectivity}), \texttt{redis} (\texttt{PING}), \texttt{mood} (\texttt{mood\_ready}), and \texttt{status} (\texttt{ok} or \texttt{degraded}). Chat re-checks Neo4j and mood readiness at request time so an unhealthy process fails closed with 503.
